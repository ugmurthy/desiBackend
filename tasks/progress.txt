## Codebase Patterns
- Use `type` imports for TypeScript types with verbatimModuleSyntax enabled (e.g., `import type { FastifyPluginAsync }`)
- Run `bun run typecheck` to verify TypeScript correctness
- SQLite: Use Bun's built-in `bun:sqlite` module for database operations
- Admin DB path: `~/.desiAgent/admin.db`
- Tenant DB path: `~/.desiAgent/tenants/{tenant_id}/agent.db` - use `getTenantDbPath(tenantId)` from user-schema.ts
- Use bcrypt for password/API key hashing (cost factor 12)
- All timestamps stored as Unix timestamps (seconds)
- User schema initialization: Use `initializeTenantUserSchema(tenantId)` which handles directory creation and returns the DB
- resource_ownership table lives in tenant's agent.db alongside dags/executions/sub_steps
- Use SQL JOINs for list operations rather than fetch-then-filter (performance)
- userId comes from `request.auth.user.id` set by authenticate middleware
- tenantDb comes from `request.auth.tenantDb` for direct DB access
- For list filtering, use getOwnedResourceIds() then filter after SDK fetch (SDK doesn't support JOIN)
- Telegram gateway schemas: tenant tables in `telegram-schema.ts` → `initializeTelegramSchemas(db)`, admin table → `initializeProfileRegistrySchema(db)`
- Pre-existing typecheck errors in client-scripts/, desiClient/, scripts/ — not from gateway code
- OTP service: `src/services/telegram-otp.ts` — issueOtp(), verifyOtp(), completeVerification(), identity CRUD helpers
- OTP uses SHA-256 hashing (not bcrypt) for fast verification of short-lived codes

- Created `src/services/telegram-otp.ts` with full OTP issuance and verification flow
- OTP generation: 6-digit code, SHA-256 hashed, stored in telegram_sessions
- OTP validity: 10 minutes (OTP_TTL_SECONDS), max 3 failed attempts (OTP_MAX_ATTEMPTS)
- Active sessions are invalidated before new OTP issuance (prevents stale sessions)
- Identity state transitions: awaiting_email → awaiting_otp → verified
- On success: links Telegram identity to backend user via email lookup in users table
- On failure: clear error messages with remaining attempts count; resets to awaiting_email on exhaustion
- Sends OTP via existing email service (DesiAgentEmailProvider)
- Files changed: src/services/telegram-otp.ts
- **Learnings for future iterations:**
  - Use SHA-256 for OTP hashing (not bcrypt) — OTPs are short-lived and need fast verification
  - Always invalidate previous active sessions before issuing new OTP
  - Use non-null assertion (!) for Uint8Array access with strict TypeScript
